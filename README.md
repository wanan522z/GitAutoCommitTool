# Git Auto Commit Tool

`Git Auto Commit Tool` 是一个给 Windows 用户使用的桌面小工具，用来帮你对本地 Git 项目做周期性自动提交，并在需要时手动推送到 GitHub。

它适合这些场景：

- 你想给正在开发的项目定时留存快照，减少忘记提交的情况
- 你不想频繁敲 Git 命令，希望用图形界面完成常见操作
- 你想把本地仓库状态、提交记录、待推送数量放在一个窗口里查看

注意：这个工具执行的是真实 Git 命令，请先确认你选择的是正确项目。

## 主要功能

- 选择本地项目文件夹并检测是否为 Git 仓库
- 一键执行 `git init`
- 设置当前仓库的 `user.name` 和 `user.email`
- 自动补充或更新 `origin` 远程地址
- 按设定间隔执行 `git add -A` 和 `git commit`
- 手动执行 `Commit`
- 手动执行 `Push`
- 查看最近提交记录和当前分支状态
- 切换到历史提交版本，并支持取消本次切换
- 查看运行日志和最近几次 push 结果

## 运行要求

- Windows 10 或 Windows 11
- 已安装 Git，并且命令行可以执行 `git --version`
- 如果运行源码，建议使用 Python 3.11 或更高版本

## 如何启动

### 方式 1：直接运行源码

```powershell
python git_auto_commit_gui.py
```

### 方式 2：运行打包好的 EXE

直接打开：

```text
dist/GitAutoCommitTool.exe
```

## 推荐放置方式

建议先自己新建一个专门的工具文件夹，再把 `GitAutoCommitTool.exe` 放进去运行，例如：

```text
D:\Tools\GitAutoCommitTool\
```

推荐这样做的原因：

- 程序的配置文件会和 `exe` 放在同一个文件夹里
- 程序的日志文件也会和 `exe` 放在同一个文件夹里
- 后续备份、移动、删除都更直观
- 不容易和别的工具文件混在一起

不建议直接放在系统目录、压缩包临时目录，或者你不方便写入的目录里。

## 第一次使用

1. 先新建一个工具文件夹。
2. 把 `GitAutoCommitTool.exe` 放进这个工具文件夹里。
3. 双击运行工具。
4. 点击“浏览”选择你的项目文件夹。
5. 如果这个文件夹还不是 Git 仓库，点击 `git init`。
6. 如果你需要推送到 GitHub，在“GitHub 链接”里填写仓库地址。
7. 填写 Git 用户名和邮箱，然后点击“保存身份”。
8. 需要立即生成一次提交时，点击 `Commit`。
9. 需要把本地提交同步到远程时，点击 `Push`。
10. 需要定时自动提交时，打开 `Auto Commit` 开关，并设置提交间隔。

## `git init` 做了什么

当你点击工具里的 `git init` 时，本质上就是在当前项目文件夹里执行一次 Git 自带的初始化命令。

它会做这些事：

- 在当前文件夹下创建 `.git` 目录
- 把这个文件夹变成一个本地 Git 仓库
- 让后续的 `Commit`、`Push`、提交记录查看等功能可以正常使用
- 如果当前目录里还没有 `.gitignore`，工具会顺手帮你创建一个空白 `.gitignore`

它不会做这些事：

- 不会自动创建 GitHub 仓库
- 不会自动上传代码到远程
- 不会自动生成第一次 commit
- 不会自动帮你写好 `.gitignore` 内容
- 不会自动设置 Git 用户名和邮箱

简单理解：`git init` 只是把“普通文件夹”变成“可被 Git 管理的文件夹”，后面的身份设置、提交、推送仍然需要你继续操作。

## 界面说明

- 项目文件夹：要操作的本地仓库目录
- GitHub 链接：用于设置或更新 `origin`
- Git 用户名 / Git 邮箱：写入当前仓库的 Git 身份
- 间隔：自动提交的执行周期，单位为秒
- 自动提交格式：固定为 `auto snapshot 编号 提交时间`
- Commit：立即执行一次本地提交
- Push：把本地提交推送到远程仓库
- 刷新：重新读取仓库状态
- 日志：查看程序运行日志
- 编辑 `.gitignore`：快速打开当前仓库的 `.gitignore`

## Auto Commit 的工作方式

开启后，程序会按你设置的时间间隔执行以下逻辑：

1. 执行 `git add -A`
2. 检查是否真的有暂存改动
3. 如果有改动，再执行 `git commit`

自动生成的提交信息格式类似：

```text
auto snapshot 12 2026-06-28 16:30:00
```

说明：

- 没有新改动时，不会产生空提交
- 遇到 Git 正在 merge、rebase 或其他占用状态时，会跳过本轮自动提交
- `Auto Commit` 默认只做本地提交，不会自动推送到 GitHub

## Push 说明

- 如果你填了 GitHub 链接，但仓库还没有 `origin`，程序会自动帮你添加
- 如果 `origin` 已存在但和输入地址不同，程序会自动更新
- 第一次 push 可能会要求你先在本机完成 GitHub 认证
- 如果你当前不在正常分支上，程序会尽量创建一个恢复分支后再推送

## 配置文件和日志

程序默认会把下面这些文件放在 `exe` 所在的工具文件夹里：

- `git_auto_commit_gui_config.json`
- `git_auto_commit_gui.log`

这意味着：

- 你把整个工具文件夹复制给别人时，程序文件、配置、日志会放在一起
- 你自己备份或迁移工具时更方便
- 只要换一台机器重新放到新文件夹里，也更容易定位配置文件

如果你是从旧版本升级过来的，新版本第一次启动时仍会尽量读取旧位置里的配置。

## 常见问题

### 1. 点了 Commit 没反应

可能是当前没有新改动可提交。程序不会生成空提交。

### 2. Push 失败

常见原因：

- 远程仓库比本地更新，需要先 `pull` 或处理冲突
- GitHub 认证未完成
- 当前仓库权限不足
- 当前目录不是安全目录，需要手动加入 `safe.directory`

### 3. 选了文件夹却提示不是 Git 仓库

说明该目录下还没有 `.git` 文件夹。你可以先点 `git init`，或者确认是否选错了目录。

## 重新打包 EXE

如果你要重新生成可执行文件：

```powershell
pip install -r requirements-build.txt
pyinstaller GitAutoCommitTool.spec
```

打包结果默认在 `dist/` 目录。

## 使用建议

- 第一次建议先拿测试仓库试一下流程
- 自动提交虽然方便，但不等于备份系统，重要项目仍建议及时 push
- 如果你的项目有编译产物、缓存文件或大文件，请先写好 `.gitignore`
