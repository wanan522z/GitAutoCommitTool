# Git Auto Commit Tool

一个面向 Windows 桌面的本地 Git 小工具，核心目标只有三件事：

1. 方便地把普通文件夹初始化成 Git 仓库。
2. 方便地手动整理本地 Commit，并在合适的时候 push 到远程。
3. 用自动快照 Commit 持续备份当前全部修改，减少忘记提交导致的丢改动风险。

它更像一个“本地 Git 操作台 + 自动快照备份器”，而不是完整的 Git 平台替代品。

## 适合什么场景

- 你有很多本地项目，想快速 `git init`、绑定远程、开始管理版本。
- 你平时不想频繁打开命令行，但又希望手动 Commit 和 Push 这两个动作足够顺手。
- 你担心开发中途忘记提交，希望程序定时把当前全部改动做成快照 Commit 留底。
- 你想在一个窗口里查看仓库状态、最近 Commit、待 push 数量和运行日志。

## 产品定位

这个工具现在的主线应该这样理解：

- `git init` 是起点：先把文件夹变成可用仓库。
- `手动 Commit + Push` 是正式版本管理：用于你主动整理阶段成果、写清晰提交说明、同步到远程。
- `Auto Commit` 是自动快照备份：用于把开发过程中的零散改动先安全留在本地历史里。

推荐心智模型：

- 手动 Commit：给“准备保留的阶段成果”命名。
- Auto Commit：给“正在进行中的全部修改”做连续快照。
- Push：把你确认好的本地历史同步到远程仓库。

## 当前核心能力

### 1. 快速创建和整理 Git 仓库

- 选择本地项目文件夹。
- 一键执行 `git init`。
- 配置或更新 `origin` 远程地址。
- 保存当前仓库的 Git 用户名和邮箱。
- 快速打开或新建 `.gitignore`。

### 2. 方便地手动 Commit 和 Push

- 图形界面手动填写 Commit 标题和正文。
- 内置常见 Commit 前缀，如 `fix`、`feat`、`refactor`、`docs`。
- 支持自定义前缀。
- 一键执行本地 Commit。
- 一键 Push 当前分支到 `origin`。
- 如果当前没有 `origin`，会根据填写的远程地址自动补上。

### 3. 自动提交快照用于备份全部修改

- 按设定时间间隔自动执行 `git add -A`。
- 只有检测到真实改动时才创建 Commit，不制造空提交。
- 自动快照默认只提交到本地，不自动 Push。
- 遇到 `merge`、`rebase`、`cherry-pick` 等特殊状态时会跳过，避免打断仓库流程。

自动生成的快照消息类似：

```text
auto snapshot 12 2026-06-28 16:30:00
```

## 一个推荐用法

### 阶段一：先把目录变成仓库

1. 选择项目文件夹。
2. 如果还不是 Git 仓库，点击 `git init`。
3. 按需填写远程仓库地址、用户名、邮箱。
4. 如果项目里有缓存、构建产物或大文件，先整理 `.gitignore`。

### 阶段二：开发过程中开着自动快照

1. 打开 `Auto Commit`。
2. 设置合适的时间间隔。
3. 让工具持续把当前全部修改做成本地快照 Commit。

这样即使你中途忘了手动提交，很多过程性改动也仍然留在 Git 历史里。

### 阶段三：需要正式留档时手动 Commit

1. 点击 `Commit`。
2. 用清晰的标题整理这次阶段成果。
3. 需要时补充正文说明。

手动 Commit 更适合表达“这次改动完成了什么”，而不是简单代替快照。

### 阶段四：确认后再 Push

1. 检查最近 Commit。
2. 确认当前分支和远程地址无误。
3. 点击 `Push` 同步到远程。

## 手动 Commit 的行为说明

手动 Commit 窗口支持：

- 选择常见前缀。
- 自定义前缀。
- 填写标题。
- 可选填写正文。

生成结果类似：

```text
fix: 修复手动提交后列表未刷新的问题

补充说明可以写在这里，作为 commit body 保存。
```

如果当前没有新的文件改动，但仓库里已经存在最近一次提交，工具不会强行创建空 Commit，而是把最近一次 Commit 改名为你这次填写的新标题和正文。

这意味着：

- 手动 Commit 既可以创建新提交。
- 也可以在“本轮其实没新改动”的情况下，帮你重新整理最近一次提交信息。

## Auto Commit 的边界

Auto Commit 的目标是“本地快照备份”，不是“自动发布”。

它不会替你做这些事情：

- 不会自动创建 GitHub 仓库。
- 不会自动完成 GitHub 登录或鉴权。
- 不会自动执行 `pull`、`merge`、`rebase`。
- 不会自动解决冲突。
- 不会默认自动 Push 到远程。

所以更合适的使用方式是：

- 用 Auto Commit 保住过程性改动。
- 用手动 Commit 整理清晰历史。
- 用 Push 在你确认后再同步远程。

## Commit 记录与仓库状态

界面里可以直接看到这些信息：

- 当前分支状态。
- 工作区是否有未提交改动。
- 待 push 的 Commit 数量。
- 最近 Commit 记录。
- 运行日志。

Commit 记录支持按范围查看：

- `全部分支`
- `当前分支`
- 其他本地分支

这对下面两类场景比较有用：

- 你只想看当前开发线，不想被别的分支历史干扰。
- 你需要从全局视角回看本地所有分支上的提交记录。

## Push 相关说明

- 如果填写了远程地址，但仓库里还没有 `origin`，工具会自动添加。
- 如果 `origin` 已存在但地址不同，工具会自动更新。
- 第一次 Push 可能仍需要你先在本机完成 GitHub 鉴权。
- 如果当前处于 `detached HEAD`，工具会优先尝试创建恢复分支后再继续 Push。

常见 Push 失败原因：

- 远程分支比本地更新，需要你先手动处理同步关系。
- 本机 GitHub 凭据或权限有问题。
- 当前目录被 Git 判定为 `safe.directory` 之外的不安全目录。
- 当前仓库还没有任何提交。

## 启动方式

### 直接用 Python 运行

```powershell
python git_auto_commit_gui.py
```

### 运行已打包的 EXE

```text
dist/GitAutoCommitTool.exe
```

建议把 EXE 放在单独工具目录里使用，例如：

```text
D:\Tools\GitAutoCommitTool\
```

程序运行后，会把配置文件和日志文件默认放在 EXE 所在目录，方便一起迁移和备份。

## 配置与日志文件

工具默认会在程序目录附近使用这些文件：

- `git_auto_commit_gui_config.json`
- `git_auto_commit_gui.log`

好处是：

- 工具、配置、日志更容易一起备份。
- 迁移到新机器或新目录时更直接。
- 出问题时更容易定位当前使用的是哪套配置。

## 常见提醒

### 1. Auto Commit 很方便，但不等于远程备份

Auto Commit 默认只是本地 Commit。

如果机器损坏、磁盘丢失或仓库目录被误删，仅有本地快照仍然不够。重要项目还是应该定期 Push 到远程。

### 2. `.gitignore` 很重要

如果项目里包含下面这些内容，建议尽早整理 `.gitignore`：

- 构建产物
- 缓存文件
- 临时文件
- 大体积依赖或生成物
- 不适合进版本库的本地配置

否则自动快照会把这些内容一起纳入提交历史。

### 3. 这是 Git 工具，不是云端协作平台

它主要解决的是：

- 本地仓库初始化
- 本地提交整理
- 本地快照备份
- 手动 Push 到远程

它不负责替代 GitHub、GitLab 或完整团队协作流程。

```powershell
pip install -r requirements-build.txt
python -m PyInstaller GitAutoCommitTool.spec
```
